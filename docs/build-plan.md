### G0 — Скелет: два крейта, границы, CI

**Выход.** Cargo-воркспейс: `logic/` и `canister/`. `config/testnet.toml`, `config/mainnet.toml` по схеме game-spec §11. `dfx.json`. CI: fmt, clippy, test и структурные линты ниже.

**Структурные линты (это и есть суть этапа).**
1. `cargo tree -p auction-logic --edges normal` печатает **только сам крейт**. Ноль зависимостей.
2. `grep -rE 'ic_cdk|std::(fs|net|time)|reqwest' logic/src/` → пусто.
3. `grep -riE 'solana|ed25519|sha256|candid' logic/src/` → пусто. Логика не знает про чейн и криптографию.
4. `grep -riE 'transfer|approve|splitter|treasury' canister/src/` → пусто; `grep -r 'http_request' canister/src/` → пусто. Канистра не двигает деньги; чтение чейна — только вызов SOL RPC-канистры из конфига, не outcalls.
5. Парсер `.did`: update-методы ⊆ `{create_auction, get_resolver, register_entry, accept_lot, return_lot, return_entry, cancel_auction, ready, vote, operator_refund_lot, operator_refund_entry, request_signature}`. Любой сверх списка роняет CI; полный список обязателен к G3.
6. `grep -r 'caller' canister/src/` → пусто. Авторизация — подпись кошелька, не principal.
7. Линт профилей: mainnet-профиль, содержащий `Custom`, роняет сборку (правило ядра, core-spec §8).

**DoD.** Дерево совпадает с картой в `CLAUDE.md`. Линты зелёные на пустом репозитории. Ни одного сетевого значения, адреса или principal в коде.

### G1 — `logic` — машина, финал, голоса

**Вход.** game-spec §5, §9, §10.

**Выход.** Крейт `auction-logic`: машина состояний `(State, Action, now) → Result<Transition>`; финал торгов — `winner(lots) -> Option<LotRef>` над снапшотом реестра (суммы живых вкладов, `seq` последней живой регистрации); подсчёт голосов (строгое большинство, переполнение → `cancel`); правило дедлайна §9; `LOGIC_VERSION`. Время — аргумент. Ни адресов, ни хешей: лоты и вклады — непрозрачные id и числа.

Заголовок `lib.rs`:
```rust
#![forbid(unsafe_code)]
#![deny(clippy::unwrap_used, clippy::expect_used, clippy::panic,
        clippy::arithmetic_side_effects, clippy::indexing_slicing)]
```

**DoD.** Property-тесты (`proptest`, dev-dependency) доказывают:
- машина: недостижимые переходы недостижимы; время применяется раньше действия; вердикт единственен и терминален;
- финал: победитель — максимум сумм живых вкладов среди принятых; ничья — меньший `seq` последней живой регистрации; возвраты меняют и сумму, и `seq`-разрешение; нулевые суммы и пустое множество → победителя нет; детерминизм — перестановка реестра не меняет исход;
- голоса: `settle` ⇔ Σdone > Σnot_done; ничья/пусто → `cancel`; переполнение суммирования → `cancel`, не паника;
- дедлайн: `ok` ⇔ `deadline ≥ created + duration + perform_window + voting_period + margin`, checked, переполнение → ошибка.
- `LOGIC_VERSION` определён и покрыт тестом на неизменность.

### G2 — Канистра: торги — реестр, чтение чейна, возвраты

**Вход.** game-spec §2, §4, §6, §7, §8 (путь вкладчика), §11.

**Выход.** Канистра поверх `logic`: `create_auction` (подпись КМ, UTF-8 сообщения §6, пиннутые вектора в `auth.rs`), `get_resolver` (бесправный, stateless-деривация `[lot_id]`), `register_entry` (деривация + пересборка соли `crown-salt` + адрес `crown-derive` + **чтение чейна** через клиент SOL RPC-канистры по паттерну `crown-core/index/src/source/solana.rs` + запись с `seq`), `accept_lot`, `return_lot`, `return_entry`, `cancel_auction` (подписи КМ), стабильная память, сертифицированные query. В PocketIC SOL RPC — **мок-канистра** с candid-интерфейсом реальной (паттерн mock-index соседей), отдаёт заготовленные аккаунты.

**DoD.**
- `auction_id`, `lot_id`: кросс-вектора python3 hashlib ↔ канистра.
- Резолверы двух лотов различны и стабильны; паритет пути с `schnorr_public_key` доказывается подписью в G3 — извне канистры дериватный ключ не пересчитать (path включает canister id).
- Регистрация: честный аккаунт из мока принимается; матрица отказов — нет аккаунта, чужой `owner`, расхождение `donor`/`salt`, `settled = true`, чужая цена игры (fee в соли), короткий дедлайн, ниже `min_bid`, лот возвращён, аукцион не в `BIDDING`, повторная регистрация. Транспортная ошибка мока → ошибка вызова, записи нет.
- Подписи участников: чужой ключ, чужой аукцион, чужая канистра, чужой кластер, чужое действие — отвергнуты; точные строки сообщений пиннуты юнит-тестами.
- Возвраты: `return_entry` исключает вклад из суммы и `seq`-разрешения лота; `return_lot` закрывает лот для регистраций; атрибуция (кто, когда) сертифицирована.
- Апгрейд: записи побайтово целы (candid `opt` в хвостах — upgrade-safe).

### G3 — Канистра: финал, голосование, подписи

**Вход.** game-spec §5, §8, §9, §10, §13.

**Выход.** Таймеры (конец торгов, срок выполнения, конец голосования) с порционной обработкой и добором (урок «Сбора»); финал — победитель из `logic`, вердикты; `ready`, `vote` (вес из `crown-index` — мок-канистра), `operator_refund_lot` / `operator_refund_entry`; `request_signature` с трёхступенчатым разрешением исхода и подписью `sign_with_schnorr([lot_id])`, локальная верификация против резолвера лота. Вердикт пишется в стабильную память **до** первой подписи — порядок закреплён тестом.

**DoD.**
- message вердикта побайтово равен `DOMAIN ‖ program_id ‖ escrow ‖ outcome` — пиннуто на вектор.
- Трёхступенчатое разрешение: повкладовый `cancel` перекрывает `settle` лота; незнакомый лот финализированного аукциона получает `cancel`; до финала — «исхода нет»; фаззом по полям рождения.
- Один исход навсегда: повторный тик, повторный запрос, запрос с другими аргументами того же эскроу — не выдают второй исход; ретрай — эквивалентная подпись того же message.
- Финал против живого реестра: максимум/ничья/нулевые суммы/все возвращены; всплеск лотов не подвешивает таймер (порция + добор — тест с сотнями лотов).
- Голос: порог `MIN_VOTE_WEIGHT`, дедуп, отказ `crown-index` без потери голоса, голос вне `VOTING` отвергнут.
- Оператор: возврат из любого состояния до вердикта, включая `VOTING`; `settle` оператору недостижим ни одним входом; пустой `operator_wallet` в конфиге = методы отключены.
- Сертификат состояния проверяется против root key NNS. Линт `.did` — полный список G0/5 — зелёный.

### G4 — E2e через реальную сеть

**Вход.** game-spec §12.

**Выход.** E2e-скрипт в `scripts/` против devnet: живые фабрика `two-outcome` и сплиттер, локальный `crown-index`, реальная SOL RPC-канистра локальной реплики (Custom-источники testnet-профиля — паттерн e2e ядра). Драйвер транзакций в `e2e/` — по образцу соседних игр; game-агностичные куски копируются как есть (позиция владельца).

**DoD.**
- **Devnet:** аукцион → лот A (вкладчик 1) + долив в A (вкладчик 2) + лот B → accept A и B → конец торгов → A победил (сумма), B получил `cancel` и исполнен `claim(1)` → `ready` → голос «done» → `settle` → `claim(0)` обоих эскроу A → книга локального `crown-index` показывает **каждому вкладчику A** ровно его `gross_i` за вычетом комиссии игры (репутация = дошедшее до КМ), аномалий 0.
- Регистрация фантомного эскроу (валидные поля, не рождённый) отвергается **реальным** чтением; после рождения — принимается.
- `return_entry` до финала: вклад исключён, `claim(1)` проходит по повкладовой подписи.
- Подпись эскроу чужого лота отвергается реальным контрактом; `cancel` и `refund()` не эмитят `Settled` — книга не меняется.
- **`git diff` в `crown-core` и `crown-factory` пуст.** Это и есть настоящий тест шва.

После зелёного G4 игра готова в devnet-объёме: полный цикл «торги → чтение → финал → голосование → книга» работает без клиента.
